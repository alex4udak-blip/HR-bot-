# ClickUp → Энцеладус: импорт архивных кандидатов (Option 1)

- **Дата:** 2026-06-16
- **Автор:** 8ridge (+ Claude)
- **Статус:** Бэкенд реализован и провалидирован на реальном файле (109 строк). Ожидает: интеграционный прогон с БД + ревью.
- **Ветка:** `feature/factorial-module` (HR-bot), без коммита.

## Контекст

Нужно перенести накопленную базу кандидатов из ClickUp в Энцеладус/HR-bot как **архивные** записи (чтобы при повторном отклике HR видел историю). Раздел Activity (видео/PDF/переписка) — вне scope, только текстовые поля карточек.

Извлечение из ClickUp **уже решено заказчиком**: ClickUp REST API бесплатный (~100 req/min), Владимир сам выгрузил данные скриптом в CSV. Доступ к ClickUp и платный List Export не нужны. Наша задача — **парсер/импортёр CSV → `entities`**.

## Решение: Option 1 — расширение существующего CSV-импорта

Переиспользуем готовый мастер `/api/import` ([`backend/api/routes/csv_import.py`](../../../backend/api/routes/csv_import.py)) + страницу [`frontend/src/pages/CsvImportPage.tsx`](../../../frontend/src/pages/CsvImportPage.tsx). **Миграции БД не требуются:** кастом-поля идут в JSON-колонку `entities.extra_data`, статус `reserve` уже есть в enum `entitystatus` (`start.sh:322`).

Альтернативы (отклонены): отдельный модуль ClickUp-импорта (больше нового кода без выгоды на v1); разовый CLI-скрипт (нет UI, который просил заказчик).

## Источник данных (реальный экспорт)

Файл `HR_отдел_..._Performance.csv`: **109 кандидатов, 57 колонок**.

- Скрипт заказчика уже **расшифровал dropdown/labels в текст** (`Источник=hh/rabota.by`, `Статус занятости=Полностью свободен`) и распарсил даты — сырой ClickUp-кодировки (id опций, unix-мс) в CSV нет.
- Колонки: служебные (`task_id`, `funnel_space/folder/list`, `status`, `status_type`, `assignees`, `date_created`, `url`, `description`) + 38 кастом-полей с префиксом `cf:` + `comments`.
- **Email отсутствует** как класс. Контакты — `cf:Контактный номер` и `cf:Telegram` (оба 109/109).
- Реально заполнено ~24 колонки; ~30 (часть «Фарм: …», «1 год/2 недели/3 мес» и т.п.) пусты в этом листе и варьируются от вакансии к вакансии → импортёр обязан быть гибким по схеме.

### Подводные камни (найдены в данных)

1. **Дата рождения / таймзоны.** `cf:Дата рождения` сконвертирована в UTC (теряет день для рождённых 00:00–02:59 локального). Таймзоны в данных разные (`+00:00` ×72, `+02:00` ×13, `+03:00` ×23, `+04:00` ×1) — «прибавить N часов» не подходит. Берём **календарную дату до `T`** из ISO в `description` (есть 109/109). Подтверждено: 2 записи спасены от off-by-one.
2. **Локация — гео-JSON.** `cf:Местонахождение` = `{"location":{lat,lng},"place_id","formatted_address"}`. Берём `formatted_address`.
3. **Мусорные DOB.** 4/109 имеют дату рождения из будущего (форма подставила дату отправки; среди них тестовые «Кабан Кабаныч»). Защита: принимаем DOB только в диапазоне `1940 … тек.год−14`, иначе не пишем (кандидат импортируется без DOB).
4. **Потеря данных в текущем импортёре.** Фронт шлёт все заголовки в маппинге → `unmapped_csv_cols` всегда пуст → всё, что не легло в first-class поле, **терялось**. Исправлено: любая колонка, не привязанная к first-class полю, сохраняется в `extra_data` под своим именем.

## Изменения в коде

### `backend/api/routes/csv_import.py`
1. Авто-детект ClickUp по сигнатуре (`task_id` + `funnel_*`) → ClickUp-ветка нормализации.
2. Дефолтный статус для ClickUp → `reserve` (если пользователь не выбрал другой).
3. Дедуп по **email / phone / telegram** (предзагрузка существующих + внутри батча). Раньше — только email.
4. Фикс потери данных: неназначенные колонки → `extra_data` (через `_FIRST_CLASS_FIELDS`).
5. `_clickup_postprocess`: birth_date (из description, с валидацией диапазона), location (formatted_address), clickup_status/task_id/url, чистка сырых ключей.
6. Авто-маппинг умнее: срез префикса `cf:` + RU-алиасы («контактный номер» → phone).
7. `funnel_list` → `position`.
8. Хелперы: `_normalize_phone`, `_extract_clickup_birthdate`, `_extract_clickup_location`.

### `frontend/src/pages/CsvImportPage.tsx`
1. Статус `reserve` (Резерв) добавлен в список статусов.
2. Подпись чекбокса дедупа → «по телефону / Telegram / email».

## Валидация (на реальном файле, без БД)

Прогон логики на 109 строках:
- imported **108**, skipped_dup **1** (реальный повтор), missing_name **0**;
- birth_date **104** (4 мусорные отсеяны), location **108**, phone **108**, telegram **108**;
- off-by-one по дате: **2** записи исправлены.

Авто-маппинг определил без ручной правки: `name`, `status`, `cf:Telegram`→telegram, `cf:Источник`→source, `cf:Контактный номер`→phone.
