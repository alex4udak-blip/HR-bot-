# Huntflow missing features

Фиксируем элементы интерфейса Huntflow, которые нужны для визуального сходства, но не имеют полноценного аналога в текущем HR-bot. Их не реализуем как функционал без отдельной задачи.

- HR sidebar: `Календарь` есть в Huntflow, отдельной HR-страницы календаря в HR-bot сейчас нет. В сайдбаре оставлен визуальный пункт с toast-заглушкой.
- Candidate actions: `СМС` в Huntflow открывает промо/подключение SMS-рассылок и напоминаний. В HR-bot SMS-модуль сейчас отсутствует, поэтому полноценный SMS-попап не добавляем без отдельной задачи.
- Candidate actions: `Форма обратной связи` и `Анкета кандидата` в Huntflow являются отдельными сценариями коммуникации/сбора данных. В HR-bot сейчас нет соответствующих модулей, поэтому кнопки дают безопасную заглушку без записи данных.
- Resume actions: для демо-резюме на HR-странице нет исходного файла, поэтому Huntflow-кнопки `Скачать` и `Распечатать` не могут выполнить файловую операцию без отдельного backend/file-flow.
- HR settings center: `Темы оформления`, `API и вебхуки`, `Почта и календарь`, `Джоб-сайты`, `Интеграции` в Huntflow являются отдельными настройками. В HR-bot нет полноценного пользовательского UI для этих разделов, поэтому в центре настроек они показаны как недоступные без добавления нового функционала.
- HR sidebar plus menu: `Добавить событие` есть в меню Huntflow, но отдельного глобального создания HR-события без выбранного кандидата/воронки в HR-bot сейчас нет. Пункт оставлен визуально с toast-заглушкой.
- Vacancy form: `Настроить поля формы`, `Сколько человек нужно нанять`, дедлайн найма, `Этапы подбора` как отдельная настройка формы, `Шаблон оффера` и прикрепление файлов к вакансии в Huntflow являются отдельными настройками вакансии/воронки. В HR-bot нет отдельных моделей для этих параметров/файлов вакансии, поэтому они переносятся визуально или мапятся на ближайшие существующие поля без добавления нового backend-функционала.
- Vacancy request preview: кнопка Huntflow `Не брать в работу` требует отдельного сценария отказа от назначенной заявки. В HR-bot сейчас есть только `Взять в работу`, поэтому кнопка оставлена визуально с безопасной toast-заглушкой без изменения данных.
- New candidate modal: `Импорт из почты` в Huntflow открывает импорт кандидата из подключенной почты. В HR-bot нет отдельной интеграции почтового импорта кандидатов, поэтому кнопка переносится визуально без добавления нового backend-функционала.
- New candidate modal: `Загрузка из файла -> Из xlsx` в Huntflow ведет к отдельному импорту кандидатов из таблицы. В HR-bot сейчас есть парсинг резюме через существующий parser flow, но нет отдельного xlsx-import сценария для нового кандидата.
- Vacancy detail: Huntflow предлагает браузерное расширение/инструкцию для сохранения найденных на джоб-сайтах резюме одним кликом. В HR-bot такого расширения и backend-flow сейчас нет, поэтому в карточке вакансии оставлена безопасная заглушка.
- Vacancy detail: меню `options-main/options-2` в Huntflow содержит дополнительные настройки отображения/действия вакансии. В HR-bot сейчас есть редактирование/закрытие/удаление через существующие формы, но нет отдельного Huntflow-меню с полным набором действий.
- Closed vacancies search: в Huntflow строки закрытых вакансий могут показывать заказчиков и кандидатов с `Оффер принят`. Текущий список вакансий HR-bot не отдаёт эти агрегированные связи в vacancy overview, поэтому экран использует только реальные поля вакансии: даты, департамент, рекрутер и количество кандидатов.
- Closed vacancies filters: в Huntflow меню `Фильтры` для закрытых вакансий поддерживает расширенные поля поиска по заказчикам/офферам и другим агрегатам. В HR-bot сейчас нет этих полей в API списка вакансий, поэтому реализован только безопасный попап со сбросом доступных фильтров без добавления фиктивных данных.
- All candidates top stages: в Huntflow есть отдельные этапы `Принятие решения/ СБ`, `Вышел на работу`, `Резерв` и онбординг-этапы после оффера. В текущей модели HR-bot есть только реальные статусы `new`, `screening`, `practice`, `tech_practice`, `is_interview`, `offer`, `hired`, `rejected`; поэтому неподдержанные Huntflow-этапы показаны как недоступные, без фальшивого маппинга на чужие данные.
