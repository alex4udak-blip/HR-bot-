# Расширение: надёжный парсинг LinkedIn и Habr Career

Дата: 2026-06-17
Подход: **C — Гибрид (JSON-first где есть → надёжный DOM → универсальная санитизация)**

## Проблема

Парсеры LinkedIn и Habr в расширении (`backend/chrome-extension/content/`) уже
существуют и зарегистрированы (manifest v1.6.4, флоу `popup.js`/`background.js`
полностью source-agnostic), но дают мусор или пустоту. Подтверждено по коду и по
реальным образцам страниц:

1. **Habr — «грязный парсинг».** `habr.js` использует широкие селекторы
   `[class*="age"]`, `[class*="salary"]`, `[class*="gender"]`, `[class*="contact"]`.
   `[class*="age"]` цепляет любой элемент с «age» в классе (`page`, `language`,
   `image`…) → в поле «Возраст» падает целый блок навигации. `salary`-fallback
   гонит regex по `document.body.innerText` и ловит встроенный JSON. Контакты
   собираются нестрого (ловится служебный `habrcareer_bot` как телеграм кандидата).
2. **LinkedIn — «слепой парсинг».** `linkedin.js` завязан на `.text-heading-xlarge`
   и `.pv-text-details__*`. В реальном DOM эти семантические классы **алиасятся в
   обфусцированные** (`.text-heading-xlarge,.UFIQjOzdKkJcDZJMGMhakydpxeWZnuAUuQbI{…}`
   в CSS — у живого элемента только обфусцированный класс), а fallback на `h1`
   мёртв: **в сохранённом профиле 0 тегов `<h1>`**. Имя не находится → `full_name`
   пустой → popup показывает «Не нашёл данных».

### Что показали реальные образцы
- **LinkedIn** (сохранённый профиль, 2.7 МБ): нет `ld+json`, нет инлайн-Voyager-JSON
  (`included`/`firstName` = 0). Имя есть в `<title>` («Сабрина Ахмедова | LinkedIn»).
  Структура: **0 `<h1>`, 7 `<h2>`, 8 `<section>`, 48 `aria-label`**. Значит данные
  достаём из DOM по устойчивым якорям, не по классам.
- **Habr**: профиль (включая контакты, навыки, опыт, локацию) отдаётся **встроенным
  JSON** на залогиненной странице (см. структуру: `age`, `experience`, `location`,
  `skills[].title`, `contacts.items[]` по `kind`). Анонимный запрос упирается в
  login-wall — нужен **залогиненный** сохранённый HTML, чтобы зафиксировать путь к JSON.

## Архитектура

Парсеры остаются изолированными по сайтам; добавляется общий слой-предохранитель.

### Новый модуль `content/common.js` (инжектится первым)
Экспортирует в неймспейс `window.__ENC__`:
- `extractProfileJson(matchers)` — ищет встроенный JSON-остров: `<script
  type="application/json">`, известные глобалы (`window.__INITIAL_STATE__` и т.п.),
  инлайн-данные; возвращает распарсенный объект или `null`.
- `sanitizeRecord(data)` — финальная очистка всех полей (см. ниже).
- `validators` — `isEmail`, `isPhone`, `normalizeTelegram`.
- `cleanText(s, maxLen)` — trim + схлопывание пробелов + обрезка.

### Изменения по файлам
- **`content/habr.js`** (переписать): JSON-first (профильный JSON) → строгий
  DOM-fallback (точные селекторы, **без** `[class*=…]`) → `sanitizeRecord`.
- **`content/linkedin.js`** (переписать): имя из `document.title` (срезаем
  ` | LinkedIn` и ведущий `(N)`); опыт/навыки/образование/языки — обходом
  `<section>` с классификацией по тексту `<h2>`; должность/локация — топ-карточка
  / `<meta property="og:*">`; **hydration-wait** (MutationObserver + ретраи, пока
  не появится имя/секции, т.к. SPA догружает DOM асинхронно — это и есть причина
  «Не нашёл данных» вживую); → `sanitizeRecord`; мягкая деградация (если профиль
  не дался — хотя бы имя + URL, `name_is_placeholder`, рекрутёр дозаполняет руками).
- **`content/hh.js`** (минимально): работает — просто прогнать вывод через
  `sanitizeRecord` для подстраховки. Логику парсинга не трогаем.
- **`manifest.json`**: `content/common.js` первым в `js`-массиве каждого
  `content_scripts`; бамп версии 1.6.4 → 1.6.5.
- **`background.js`**: программный инжект грузит пару `['content/common.js',
  '<site>.js']` (иначе fallback на нестандартных URL потеряет общий модуль).

Бэкенд менять **не нужно**: `popup.js` и `/api/magic-button/parse` уже принимают
все поля для любого `source`. Опционально (2-й рубеж, рекомендуется) — лёгкая
серверная санитизация входящих полей в `magic_button.py`.

## Слой санитизации (главный предохранитель)

`sanitizeRecord` вызывается **перед** `chrome.runtime.sendMessage({type:'PARSE_RESULT'})`,
поэтому мусор физически не доходит до popup/CRM. Правила:
- Строки: trim + схлопывание пробелов + обрезка по лимиту поля (имя ≤80, должность
  ≤120, город ≤60, возраст ≤20, ЗП ≤60, компания ≤120, опыт-саммари ≤2000, элемент
  навыка/языка ≤60).
- **Обнуляем значение целиком**, если содержит `{`, `}`, `<`, `>`, `function`,
  `var `, `window.`, `=>` или выглядит как JSON-простыня.
- Эвристика нав-мусора (маркеры Habr: «Все сервисы», «Сообщество IT», «Создать
  вакансию», «Кабинет компании», «Подключи Бустер») → пусто.
- Массивы (skills/languages): санитайз каждого элемента, мусорные выкидываем.
- Контакты: `email`/`phone`/`telegram` только если проходят валидатор (отсекает
  служебный `habrcareer_bot` и подобное).

## Поток данных (без изменений плумбинга)

content-скрипт (JSON-first → DOM-fallback → `sanitizeRecord`) → `PARSE_RESULT` →
`background.js` (storage) → `popup.js` `GET_PARSED_DATA` → показ → «Добавить» →
`POST /api/magic-button/parse` (поля и `source` уже динамические).

## Обработка ошибок и деградация
- JSON не найден → DOM-fallback.
- DOM пустой/не прогрузился (LinkedIn SPA) → hydration-wait с таймаутом; если всё
  равно пусто — минимальный набор (имя из title + URL) вместо «всё пропало».
- Поле не прошло санитизацию → пустое (popup уже показывает ручной ввод для
  пустых контактов/имени-плейсхолдера).

## Тестирование
- **Юнит (Node)** на чистых функциях: `sanitizeRecord` (кейсы: JSON-блоб → пусто,
  нав-текст → пусто, нормальное имя → проходит), мапперы JSON→поля на фикстурах
  реальных образцов (LinkedIn DOM-фикстура есть; Habr — после залогиненного образца).
- **Ручной чек-лист**: открыть профиль Habr + LinkedIn → popup показывает чистые
  структурированные поля (без `{}`/нав-мусора) → «Добавить» → карточка в CRM чистая.

## Деплой
Бамп версии → бэкенд-redeploy на Saturn (Coolify Deploy вручную — автодеплой не
работает; отдаёт свежий zip из `backend/chrome-extension/`) → пользователь и Мария
**пере-скачивают и переустанавливают** расширение (unpacked → автообновления нет;
проверить версию в футере popup = 1.6.5).

## Открытая зависимость (для имплементации)
LinkedIn-образец получен (`Сабрина Ахмедова _ LinkedIn.html`). Для Habr нужен
**залогиненный** сохранённый HTML профиля `career.habr.com/lebedevnet` (Ctrl+S →
«веб-страница полностью»), чтобы зафиксировать точный путь к встроенному JSON.
Без него Habr-JSON-маппинг пишется по структуре со скриншота + проверяется на
живой странице при имплементации.

## Вне скоупа (YAGNI)
- Перехват XHR/Voyager-запросов LinkedIn (сложно из content-скрипта; инлайн-JSON
  на этой странице отсутствует — обходимся DOM).
- Серверный ре-парсинг/обогащение (отдельная задача, если понадобится).
- Поддержка других источников кроме hh/habr/linkedin.
