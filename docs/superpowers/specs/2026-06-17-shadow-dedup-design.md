# Модуль теневой дедупликации кандидатов

Дата: 2026-06-17
Статус: дизайн на ревью

## 1. Цель и поведение
**Создаём** «теневую» (архивную) базу кандидатов — сейчас её НЕТ. Архивные
кандидаты не видны в активных списках, канбане, поиске. **Видеть и управлять
архивом может только суперадмин** — через шестерёнку (меню настроек орги, где
лежит «Факториал»). При добавлении НОВОГО кандидата (вручную или через расширение)
система в фоне сверяет его с архивом по контактам. При совпадении новый кандидат
создаётся как обычно, но получает флаг-триггер: в карточке сверху — оранжевый
баннер «Похожий кандидат есть в базе» с кнопкой «Проверить», открывающей
сплит-сравнение с тремя действиями (объединить / разные люди / закрыть).

## 2. Что переиспользуем и что чиним (из аудита)
**Переиспользуем как есть:**
- `similarity.py`: `normalize_phone()` (чистка спецсимволов, 8→7), `normalize_email()`, `detect_duplicates()` (имя с транслитом + email + телефон).
- Существующую живую фичу дубликатов: `GET /entities/{id}/duplicates`, `POST /entities/{id}/merge`, `GET /entities/{id}/compare/{other}`, фронт `DuplicateWarning.tsx` (баннер), `SimilarCandidates.tsx` (модалка сравнения). Баннер и сплит-модалку строим ПОВЕРХ них.

**Чиним (необходимо для merge):**
- `EntityStatus.merged` отсутствует в enum → merge падает. Добавить значение (через `start.sh ALTER TYPE`, как делали reserve/withdrawn).
- merge не перецепляет на основную сущность: `stage_transitions`, `entity_analyses`, `entity_criteria`, `entity_ai_conversations`, tag-связи, `analysis_history`. Дописать — иначе теряется история (а нам нужен единый таймлайн).
- Битый импорт `from api.services.similarity import merge_entities` в `candidate_database.py:389`.

## 3. Модель данных
- **`Entity.is_archived: bool`** (default false, индекс). Миграция — идемпотентный `ALTER TABLE entities ADD COLUMN IF NOT EXISTS is_archived` в `start.sh`.
- **Наполнение архива:**
  - bulk/парсер-импорт → `is_archived = true`: `csv_import.py`, `parser.py`, `parse_jobs.py`, `interns.py`.
  - расширение (magic-button), ручное добавление (`entities/crud.py`), формы (`forms.py`) → `is_archived = false` (активные).
  - кнопка **«В архив»** у любого кандидата → `is_archived = true` (ручной перенос в архив).
- **Фильтрация активных выборок** (`is_archived = false`) в ~8 точках: `candidate_search` (поиск+канбан+bulk), `vacancies/kanban`, `entities` list, аналитика. Детальную карту точек дал аудит.
- **Флаг дубля** — в `extra_data` нового кандидата: `hidden_duplicate_id: <id архивного>` (наличие = баннер).
- **Разрыв связи** — `extra_data.dismissed_duplicate_ids: [id, ...]` (чтобы повторный детект не вешал баннер снова).

## 3a. Доступ и расположение (только суперадмин)
Теневая база — **суперадминский раздел**, обычным HR/админам не виден.
- **Страница «Архив кандидатов»**: новый пункт в меню-шестерёнке
  `HR_SETTINGS_ORG_ITEMS` (там же, где «Факториал»; `Layout.tsx:368`), показывается
  только при `user.role === 'superadmin'`. Маршрут напр. `/candidate-archive`,
  обёрнут в супер-админ-гейт (как `RoleRoute`, но строго superadmin).
- На странице: список архивных кандидатов (`is_archived=true`) с поиском; просмотр
  карточки; (опционально) «Восстановить из архива» / удалить.
- Бэкенд: эндпоинт листинга архива — строго `superadmin`. Активные эндпоинты архив НЕ отдают.
- Кнопка **«В архив»** у кандидата доступна HR (любой может убрать кандидата в архив),
  но СМОТРЕТЬ архив списком — только суперадмин.

## 4. Детект (бэкенд)
- Триггер: при создании АКТИВНОГО кандидата (manual + magic-button). Импорт-в-архив не сверяем (он сам и есть архив).
- Сверка по жёстким контактам: нормализованный телефон ИЛИ email, **только против `is_archived = true`**, исключая `dismissed_duplicate_ids`.
- При совпадении: пишем `extra_data.hidden_duplicate_id = <archived_id>` (сильнейшее/первое совпадение).
- Ответ создания дополняется: `has_hidden_duplicate: true`, `hidden_duplicate_id`.
- Переиспользуем `normalize_phone/email`; запрос скоупим на архив.

## 5. API
- Создание кандидата → возвращает `has_hidden_duplicate` + `hidden_duplicate_id` (см. §4).
- Сравнение: переиспользуем `GET /entities/{id}/compare/{archived_id}` (данные обеих карточек для модалки).
- **«Объединить»**: расширяем/переиспользуем `POST /entities/{id}/merge` — survivor = активный НОВЫЙ, source = архивный: историю архивного перецепляем на нового (вся §2-история), архивный помечаем `merged` (soft, остаётся в БД), снимаем `hidden_duplicate_id`.
- **«Разные люди»**: `POST /entities/{id}/dismiss-duplicate` → добавляет archived_id в `dismissed_duplicate_ids`, снимает `hidden_duplicate_id`. Оба остаются (архивный — в архиве).
- **«Закрыть»**: ничего на бэке, флаг остаётся.

## 6. UI — баннер и сравнение (для добавляющего: HR/админ/суперадмин)
Полный список архива — только суперадмин (§3a). Здесь — реакция на КОНКРЕТНОЕ
совпадение при добавлении кандидата.
- **Баннер** (когда есть `hidden_duplicate_id`): широкий, оранжевый/горчичный, над ФИО+фото в карточке. Текст белым «Похожий кандидат есть в базе». Справа — белая кнопка «Проверить». Точка вставки: `AllCandidatesPage.tsx`, над action-баром карточки.
- **Сплит-модалка** (по «Проверить»): полноэкранная, тёмный фон, заголовок «Проверка похожих кандидатов», две колонки. Слева — новый кандидат (фото, ФИО, должность, ЗП, контакты), справа — архивный (та же структура). Совпадающие текстовые поля подсвечены светло-жёлтой заливкой; различающиеся — без подсветки.
- **Блок действий** (по центру внизу, кнопки друг под другом):
  - «Объединить» (зелёная) → merge → архивный исчезает, история подтянута к новому, баннер снят.
  - «Нет, это разные люди» (красная) → dismiss → баннер исчезает навсегда, оба профиля остаются.
  - «Закрыть» (контур) → закрывает модалку, баннер остаётся до решения.
- Базируемся на `SimilarCandidates`/`DuplicateWarning`.

## 7. Краевые случаи
- Несколько архивных совпадений → берём одно сильнейшее (один `hidden_duplicate_id`). Степпер «1 из N» — позже, если понадобится (YAGNI).
- merge обязан перецепить ВСЮ историю (иначе сирота-данные — аудит показал текущие пробелы).
- Архивные не должны утекать в активные списки (фильтр во всех ~8 точках; иначе «теневые» станут видимыми).
- Существующая фича дубликатов срабатывает на ЛЮБОЙ дубль; теневую логику делаем архив-aware (детект на создании скоупим на `is_archived=true`).

## 8. Тестирование
- Бэкенд (pytest): детект (новый vs архивный по телефону и по email; нормализация); merge (вся история перецеплена, архивный = merged, флаг снят); dismiss (флаг снят, повторное создание не вешает баннер); фильтрация (архивные не в активных выборках).
- Фронт: баннер появляется при флаге и исчезает после merge/dismiss; подсветка совпадений в модалке; три действия дергают нужные эндпоинты.

## 9. Миграция / деплой
- `start.sh`: `ALTER TABLE entities ADD COLUMN IF NOT EXISTS is_archived` + `ALTER TYPE entitystatus ADD VALUE 'merged'` (идемпотентно). Бэкенд-деплой обязателен (Coolify; автодеплой не работает).

## 10. Порядок реализации (по этапам, на завтра)
1. `is_archived`: колонка + наполнение на create-путях + фильтрация активных выборок (фундамент; делает «теневую базу» реальной).
2. Детект на создании + флаг `hidden_duplicate_id` + поле ответа.
3. Починка+достройка merge (enum `merged` + перецеп всей истории) + endpoint dismiss.
4. UI: баннер + сплит-модалка с подсветкой + 3 действия.
5. Суперадминская страница «Архив кандидатов» (пункт в меню-шестерёнке рядом с
   «Факториалом», гейт `superadmin`) + кнопка «В архив» у кандидата.

Каждый этап — отдельный коммит; после §1 и §3 (бэкенд+миграция) нужен Coolify-деплой.
